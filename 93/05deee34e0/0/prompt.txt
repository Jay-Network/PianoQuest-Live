## Part 1: Transcription Fix

### 1A. Fix output transcription accumulation (`server.ts`)
- Change `bufOutput = text` → `bufOutput += (bufOutput ? " " : "") + text` (append chunks)
- Same for `bufInput`
- Remove `inputAudioTranscription: {}` from Gemini config (stop the garbage)
- Keep `outputAudioTranscription: {}`

### 1B. Add Web Speech API for user input (`index.html`)
- `startSpeechRecognition()` / `stopSpeechRecognition()` — primary device
- `startSecondarySpeechRec()` / `stopSecondarySpeechRec()` — secondary device
- `SpeechRecognition` with `lang = 'en-US'`, `continuous = true`, `interimResults = false`
- Auto-restart on `onend` (both Chrome and Safari can stop unexpectedly)
- Send final results as `{ type: 'user_speech_transcript', text }` via WebSocket
- Use `window.SpeechRecognition || window.webkitSpeechRecognition` for Safari compat

### 1C. Server relay (`server.ts`)
- Handle `user_speech_transcript` in both primary and secondary handlers
- Broadcast as `input_transcript` to all room devices (existing message type, already displayed in chat)

## Part 2: Multi-Device Task Distribution

### Optimal 3-device setup:
| Device | Role | Responsibility |
|--------|------|----------------|
| **Ubuntu (Chrome)** | Primary + MIDI | Visualization, chat display, device manager, USB piano MIDI. No mic (piano bleed). |
| **iPad (Safari)** | Mic | Voice capture → Gemini PCM + SpeechRecognition for transcript. Near face, away from piano. Plays Gemini audio. |
| **iPhone (Safari)** | Spectator/backup | Second display for chat. Swap to mic if iPad has issues. |

### How it works:
- Mic device automatically runs SpeechRecognition (no separate role needed)
- SpeechRecognition and getUserMedia coexist on iOS Safari
- Transcript text flows: device → server → broadcast to all
